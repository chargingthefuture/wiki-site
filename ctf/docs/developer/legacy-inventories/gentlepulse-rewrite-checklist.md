# GentlePulse Rewrite Checklist

## Scope & Decisions
- [ ] Confirm rewrite boundary is ctf-only; platform is reference-only and must not be copied module-for-module.
  - Acceptance criteria:
    - Decision note links each retained behavior to a ctf contract or task.
    - No implementation task points to editing platform code.
- [ ] Freeze plugin scope for v1 parity.
  - Acceptance criteria:
    - Included: library browse, sort/tag filter, favorites, ratings, play tracking, announcements, settings, support, admin management, mood check trigger.
    - Excluded (unless approved): net-new features, redesign-only changes, non-critical analytics additions.
- [ ] Resolve auth posture mismatch before API build starts.
  - Acceptance criteria:
    - One approved model documented: authenticated-only, anonymous-only, or hybrid.
    - Client and server contracts both reflect the chosen model.
- [ ] Resolve admin endpoint mismatch before UI implementation.
  - Acceptance criteria:
    - Admin route map exists and matches UI query/mutation plan.
    - No unresolved TODO for admin meditations or admin announcements endpoints.
- [ ] Resolve progress contract mismatch before test authoring.
  - Acceptance criteria:
    - Decision recorded: explicit progress resource vs play-count-only behavior.
    - API test matrix updated to remove ambiguity.

## Phase Plan
- [ ] Phase 0: Contract alignment and decision lock.
  - Acceptance criteria:
    - Open decisions in this doc are either approved or explicitly deferred.
    - Contract package stubs exist for all required endpoints and payloads.
- [ ] Phase 1: Data model and shared schema.
  - Acceptance criteria:
    - Meditations, ratings, favorites, announcements schema validated in shared layer.
    - Migration and seed strategy drafted for ctf environments.
- [ ] Phase 2: Core API and policy controls.
  - Acceptance criteria:
    - Public/private guards, validation, and rate limits implemented per approved posture.
    - Admin APIs ship with authz and mutation auditing requirements.
- [ ] Phase 3: Web flows.
  - Acceptance criteria:
    - Library, settings, announcements, support, admin surfaces all wired to ctf APIs.
    - Error and empty states match product acceptance copy.
- [ ] Phase 4: Mobile parity.
  - Acceptance criteria:
    - Mobile feature coverage equals approved v1 scope.
    - Shared contracts are reused with no mobile-only API fork.
- [ ] Phase 5: Hardening and release.
  - Acceptance criteria:
    - Test and release gates pass.
    - Rollout/rollback notes completed.

## API Contract Tasks
- [ ] Define meditations list contract with sorting, tag filter, pagination, and active-only behavior.
  - Acceptance criteria:
    - Request and response schemas are versioned.
    - Sort modes include newest, most-rated, highest-rating.
- [ ] Define meditation detail and play tracking contracts.
  - Acceptance criteria:
    - Play event endpoint is idempotency-aware enough to prevent accidental double-submit inflation.
    - Playback URL handling is explicit in contract.
- [ ] Define ratings contract for 1-5 scale and aggregate recompute behavior.
  - Acceptance criteria:
    - Validation rejects out-of-range values.
    - Aggregate response shape is stable and consumed by web/mobile.
- [ ] Define favorites contracts: add, remove, list, single-item check.
  - Acceptance criteria:
    - Identifier strategy aligns with approved auth posture.
    - Removal semantics are unambiguous and tested.
- [ ] Define user announcement read contract.
  - Acceptance criteria:
    - Active + non-expired filtering is server-enforced.
    - Response includes type and expiry metadata needed by clients.
- [ ] Define admin meditation contracts (list/create/update/deactivate as approved).
  - Acceptance criteria:
    - Contract exists for every admin UI action.
    - Validation, authorization, and conflict/error semantics are documented.
- [ ] Define admin announcement contracts (list/create/update/deactivate).
  - Acceptance criteria:
    - Contract names and route placement are final.
    - Cache invalidation triggers are defined in client integration notes.
- [ ] Resolve mood integration boundary contract.
  - Acceptance criteria:
    - Document whether GentlePulse calls shared Mood endpoints directly or via adapter.
    - Eligibility cadence and response mapping are explicit.
- [ ] Resolve progress mismatch in API surface.
  - Acceptance criteria:
    - If progress exists, endpoints and schema are specified.
    - If progress is removed, tests and UI references are removed.

## Web Tasks
- [ ] Build library page with deterministic query-state handling.
  - Acceptance criteria:
    - Sort/tag/pagination/favorites-only state survives navigation refresh.
    - Announcement banner and empty states render correctly.
- [ ] Implement meditation card actions and loading/error states.
  - Acceptance criteria:
    - Play, rate, favorite actions provide clear success/failure feedback.
    - Aggregate rating refresh is visible without full page reload.
- [ ] Implement announcements page and settings page.
  - Acceptance criteria:
    - Announcements list handles zero-state and expiry filtering from API.
    - Settings persist and apply accessibility toggles immediately.
- [ ] Implement admin dashboard and admin announcements UI against finalized admin contracts.
  - Acceptance criteria:
    - No web call targets undefined endpoint paths.
    - Deactivate/update flows are confirmed with optimistic or refetch strategy.
- [ ] Integrate mood check modal trigger from approved contract.
  - Acceptance criteria:
    - Modal appears only when eligible and only once per session.
    - Safety response path is handled and tested.

## Mobile Tasks
- [ ] Implement GentlePulse library with contract-parity to web.
  - Acceptance criteria:
    - List filters and pagination match contract behavior.
    - Ratings and favorites are functionally equivalent to web.
- [ ] Implement settings and announcements on mobile.
  - Acceptance criteria:
    - Accessibility settings persistence works per platform standard.
    - Announcement display supports type and expiry metadata.
- [ ] Implement mood check integration on mobile.
  - Acceptance criteria:
    - Eligibility and submit flows mirror approved behavior.
    - Session gating logic is validated.
- [ ] Implement admin access strategy for mobile (include, limited, or excluded by decision).
  - Acceptance criteria:
    - Decision is explicit; unsupported views are intentionally blocked or omitted.
    - No orphan navigation paths remain.

## Shared/Schema Tasks
- [ ] Finalize shared types for meditations, ratings, favorites, announcements, and settings payloads.
  - Acceptance criteria:
    - Both clients consume shared types without divergence.
    - Runtime validation matches compile-time contracts.
- [ ] Implement migration plan for GentlePulse data structures.
  - Acceptance criteria:
    - Migration scripts are replay-safe in CI.
    - Seed scripts produce valid sample records for smoke tests.
- [ ] Standardize identifiers per auth decision.
  - Acceptance criteria:
    - Client identity/user identity policy is represented in schema and API guards.
    - No endpoint mixes incompatible identity assumptions.
- [ ] Add contract changelog entry for each non-trivial behavior decision.
  - Acceptance criteria:
    - Admin mismatch, auth posture, progress decision, and mood boundary are recorded.

## Security/Compliance Tasks
- [ ] Apply authentication and authorization middleware per approved posture.
  - Acceptance criteria:
    - Unauthorized access returns consistent errors.
    - Admin endpoints enforce role checks.
- [ ] Apply input validation and abuse controls on write-heavy endpoints.
  - Acceptance criteria:
    - Ratings/favorites/play endpoints have anti-abuse safeguards.
    - Validation errors are deterministic and non-leaky.
- [ ] Preserve privacy posture for anonymous/aggregated usage claims.
  - Acceptance criteria:
    - Stored telemetry fields are minimum necessary.
    - Privacy statements and data handling docs align with implementation.
- [ ] Ensure audit logging for admin mutations.
  - Acceptance criteria:
    - Create/update/deactivate actions emit audit events with actor and target identifiers.

## Testing & Release Gates
- [ ] Contract tests for all GentlePulse endpoints.
  - Acceptance criteria:

- [ ] Web integration tests for library, settings, announcements, and admin flows.

    - Tests use stable selectors agreed during rewrite.

- [ ] Mobile integration tests for core user flows.
  - Acceptance criteria:
    - Browse, rate, favorite, settings, and mood trigger behaviors are covered.
- [ ] End-to-end parity checks against approved v1 behavior list.
  - Acceptance criteria:
    - Every in-scope feature maps to a passing test or explicit manual verification record.
- [ ] Release readiness checklist.
  - Acceptance criteria:
    - Performance baseline, monitoring hooks, rollback plan, and migration validation are signed off.
