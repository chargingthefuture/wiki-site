# Directory Rewrite Checklist

## Scope & Decisions
- [ ] Confirm rewrite boundary is ctf-only; platform is reference-only and not a direct implementation source.
  - Acceptance criteria:
    - Feature decisions are mapped to ctf contracts and tasks.
    - No task requires editing platform modules.
- [ ] Freeze v1 parity scope for Directory.
  - Acceptance criteria:
    - Included: profile CRUD, public list/profile, admin claimed/unclaimed management, announcements, selector taxonomy, anti-scraping, geocoding support.
    - Excluded unless approved: net-new profile fields and extra discovery features.
- [ ] Resolve schema source-of-truth mismatch before migrations.
  - Acceptance criteria:
    - Shared schema and migration SQL are consistent and validated.
    - Malformed/legacy SQL artifacts are not carried into ctf.
- [ ] Resolve seed payload mismatch before seed implementation.
  - Acceptance criteria:
    - Decision recorded for nickname/displayNameType fields (remove, map, or add).
    - Seed scripts align exactly with approved schema.
- [ ] Resolve announcement route placement decision before API wiring.
  - Acceptance criteria:
    - Directory announcement APIs are defined in Directory-owned module boundaries.
    - No cross-module route ownership ambiguity remains.

## Phase Plan
- [ ] Phase 0: Contract and ambiguity resolution.
  - Acceptance criteria:
    - Open decisions are approved or deferred with owner/date.
    - API and schema draft published for review.
- [ ] Phase 1: Shared schema and migrations.
  - Acceptance criteria:
    - Profile, announcements, taxonomy references, and location fields are finalized.
    - Migrations and seed data pass local replay.
- [ ] Phase 2: API implementation.
  - Acceptance criteria:
    - User, public, and admin endpoints are implemented with policy controls.
    - Announcement APIs are colocated with Directory routes.
- [ ] Phase 3: Web implementation.
  - Acceptance criteria:
    - Dashboard/profile/public/admin/announcements routes all integrated.
    - Public privacy constraints are enforced in UI.
- [ ] Phase 4: Mobile implementation.
  - Acceptance criteria:
    - Mobile reaches parity for in-scope user flows.
    - Admin/mobile scope decision is explicit and implemented accordingly.
- [ ] Phase 5: Hardening and release.
  - Acceptance criteria:
    - Security tests, performance checks, and release gates are complete.

## API Contract Tasks
- [ ] Define authenticated profile contracts: get/create/update/delete.
  - Acceptance criteria:
    - Validation rules include description max, selector limits, required country, and URL validation.
    - Delete semantics and reason handling are explicit.
- [ ] Define authenticated listing contract for explorer/dashboard views.
  - Acceptance criteria:
    - Response includes fields needed for authenticated cards and links.
    - Visibility and verification semantics are stable.
- [ ] Define public list contract with anti-scraping and privacy posture.
  - Acceptance criteria:
    - Public list returns first-name-only identity projection.
    - Ordering/rate limiting behavior is documented and testable.
- [ ] Define public profile-by-id contract.
  - Acceptance criteria:
    - Non-public profiles return not found behavior.
    - Signal/contact exposure rules are explicit.
- [ ] Define taxonomy contracts: skills, sectors, job titles.
  - Acceptance criteria:
    - Selector payloads are normalized for web/mobile reuse.
    - Limit enforcement is server-side and mirrored in clients.
- [ ] Define admin profile contracts: list/create/update/assign/delete.
  - Acceptance criteria:
    - Claimed vs unclaimed rules are explicit.
    - Delete constraints for claimed profiles are enforced by contract.
- [ ] Define admin skills compatibility contract and delete behavior expectations.
  - Acceptance criteria:
    - Behavior for existing profiles on skill deletion is explicitly chosen.
    - Client warnings match backend behavior.
- [ ] Define announcements contracts (user read + admin list/create/update/deactivate) in Directory module.
  - Acceptance criteria:
    - Route placement is final and documented.
    - Expiry and active-state filtering rules are deterministic.

## Web Tasks
- [ ] Implement authenticated dashboard experience with profile existence branching.
  - Acceptance criteria:
    - No-profile and has-profile states are both complete.
    - Announcements banner appears per contract.
- [ ] Implement profile form create/edit/delete flows.
  - Acceptance criteria:
    - Selector caps, URL validation, description counter, visibility toggle all enforced.
    - Share URL behavior appears only when profile is public.
- [ ] Implement public list and public profile routes.
  - Acceptance criteria:
    - First-name-only public presentation is enforced.
    - Signal visibility remains hidden in public views.
- [ ] Implement admin profile management interface.
  - Acceptance criteria:
    - Claimed/unclaimed workflow, assignment, edit, and delete guardrails work as specified.
    - Search and pagination behavior aligns with accepted UX contract.
- [ ] Implement admin announcements interface with Directory-owned APIs.
  - Acceptance criteria:
    - Create/edit/deactivate actions work and refresh list state correctly.
    - No route call points to non-Directory module ownership.

## Mobile Tasks
- [ ] Implement authenticated profile flows on mobile.
  - Acceptance criteria:
    - Create/edit/delete and selector constraints match web and API rules.
- [ ] Implement public browse/profile views on mobile as approved.
  - Acceptance criteria:
    - Public privacy posture matches web behavior exactly.
- [ ] Implement announcements presentation on mobile.
  - Acceptance criteria:
    - Active/non-expired filtering and metadata display are consistent.
- [ ] Implement admin support on mobile per explicit scope decision.
  - Acceptance criteria:
    - If excluded, admin routes are absent and documented.
    - If included, permissions and key workflows are validated.

## Shared/Schema Tasks
- [ ] Finalize canonical Directory schema in shared package.
  - Acceptance criteria:
    - Profile, announcement, and taxonomy schemas compile and validate.
    - SQL migration artifacts are generated from the canonical model.
- [ ] Resolve and document schema/SQL mismatch.
  - Acceptance criteria:
    - Malformed legacy definitions are corrected in ctf migrations.
    - Schema review sign-off is captured.
- [ ] Resolve and document seed/schema mismatch.
  - Acceptance criteria:
    - Seed inputs use only approved fields.
    - Seed outputs pass API validation without custom exceptions.
- [ ] Define geocoding and coordinate persistence contracts.
  - Acceptance criteria:
    - Trigger conditions for geocode refresh are documented.
    - Backfill script contract exists for missing coordinates.
- [ ] Standardize shared DTOs for web/mobile/admin usage.
  - Acceptance criteria:
    - No per-client schema forks for core entities.
    - Public projection DTO is separate from authenticated DTO.

## Security/Compliance Tasks
- [ ] Enforce auth on authenticated and admin routes.
  - Acceptance criteria:
    - Unauthorized and forbidden responses are consistent and tested.
- [ ] Enforce admin + CSRF protections on admin writes.
  - Acceptance criteria:
    - Create/update/assign/delete announcement and profile writes require both controls.
- [ ] Enforce public anti-scraping controls for list/profile endpoints.
  - Acceptance criteria:
    - Rate limits and delay strategy are implemented and observable.
- [ ] Enforce privacy projection for public endpoints.
  - Acceptance criteria:
    - Public payload omits restricted contact fields.
    - Non-public record access is blocked.
- [ ] Emit audit logs for sensitive admin mutations.
  - Acceptance criteria:
    - Logs include actor, action, target id, and timestamp.

- [ ] Web integration tests for dashboard/profile/public/admin flows.

- [ ] Mobile integration tests for in-scope flows.
  - Acceptance criteria:
    - Authenticated and public behaviors are covered per final scope.
- [ ] Security and privacy test gates.
  - Acceptance criteria:

- [ ] Release gate checklist.
  - Acceptance criteria:
